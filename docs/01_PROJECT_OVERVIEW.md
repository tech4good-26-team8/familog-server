# Familog — 프로젝트 설명

시니어 디지털 격차 해소 / 가족 단위 커뮤니티 · 1-Day 해커톤

---

## 1. 무슨 서비스인가

**Familog(패밀로그)** = Family + Log. "가족의 기록"이라는 뜻의 가족 단위 커뮤니티.
목표는 **시니어의 디지털 격차 해소**다.

공식 한 줄 소개:
> AI 음성과 3D AI 캐릭터를 통해 가족의 소통과 추억을 기록하는 **AI 기반 패밀리 커뮤니케이션 플랫폼**

동작 관점 한 문장 요약:
> 가족이 사진·소식을 올리면, 그 사람의 **3D풍 캐릭터 아바타**와 **본인 목소리(보이스팩)**로 시니어에게 **읽어주고**, 시니어는 **말로 답장하는** 서비스.

---

## 2. 왜 만드나 (문제 정의)

스마트폰·SNS는 이미 가족 소통의 기본 창구지만, 시니어에게는 진입 장벽이 높다.

- 글자가 작고, 알림이 뭘 의미하는지 모른다.
- 조작이 낯설어 가족이 올린 소식을 "읽는 것" 자체가 부담이다.
- 그 결과 가족 대화에서 소외된다.

문제는 두 겹이다.

- **디지털 소외**: 복잡한 디지털 환경 자체가 시니어에게 진입 장벽이다.
- **정서적 고립**: 비대면 소통이 일상화되면서 가족 간 교류가 줄고, 시니어의 외로움과 사회적 고립이 심화된다.

**핵심 가설**
> "읽기 → 듣기", "텍스트 → 가족의 얼굴·목소리"로 바꾸면 시니어의 참여 장벽이 급격히 낮아진다.

시니어는 손주가 올린 글을 읽으려 애쓰는 대신, **손주 캐릭터를 보고 손주 목소리로 소식을 듣는다.**

---

## 3. 누가 쓰나

| 구분 | 역할 | 행동 |
|---|---|---|
| **시니어** | 수신자 + 음성 발신 (핵심 타깃) | 큰 화면에서 가족 아바타를 보고, 탭하면 그 가족 목소리로 소식을 듣는다. 답장은 말로 — 음성을 녹음하면 AI가 텍스트로 바꿔 가족에게 전달(STT). 조작 최소화. |
| **가족(자녀·손주)** | 발신자 | 사진+짧은 글을 올린다. 가입 시 최초 1회 목소리를 등록한다. |

설계 원칙: **시니어의 조작 부담을 최소화하는 것이 최우선.** 시니어는 읽고 쓰지 않는다 — **듣고 말한다.** 가족은 "올리는" 사람으로 역할을 명확히 나눈다.

---

## 4. 핵심 기능 4개

서비스의 가치는 세 가지 "생성"과 하나의 "역방향 소통"에서 나온다.

1. **가족 3D 아바타** — 가입 시 올린 사진을 3D풍 캐릭터로 변환. 시니어가 "누가 올렸는지"를 글이 아니라 얼굴로 인지한다. (기획상 장기적으로는 음성과 표정을 함께 전달하는 경험을 지향하되, 해커톤 범위는 정적 캐릭터 — 02 문서 §8-2 결정)
2. **개인 보이스팩** — 가입 시 지정 문장을 낭독하면, 그 사람의 목소리를 재현하는 보이스팩이 만들어진다.
3. **보이스 읽어주기(TTS)** — 게시글이 올라오면 그 사람의 보이스팩으로 텍스트를 낭독해준다. 시니어는 읽지 않고 듣는다.
4. **음성 답장(STT)** — 시니어가 말로 녹음하면 AI가 텍스트로 변환해 가족에게 전달한다. 시니어는 쓰지 않고 말한다.

1~3이 "가족 → 시니어" 방향, 4가 "시니어 → 가족" 방향을 열어 소통을 **양방향**으로 만든다. 나머지 기능(멤버·게시물 관리)은 이 생성물을 담고 화면에 연결하는 그릇일 뿐이다.

## 5. 사용자 경험 흐름

**가입 (가족)**
1. 가족 그룹 생성 또는 초대 코드로 참여
2. 프로필 사진 업로드 → 3D 캐릭터 아바타 생성
3. 지정 문장 낭독 → 보이스팩 생성

**게시 (가족)**
1. 사진 + 짧은 글 업로드
2. 그 사람 보이스팩으로 낭독 오디오가 자동 생성됨

**소비 (시니어)**
1. 홈에 가족 아바타 카드가 크게 뜬다
2. 게시물을 탭하면 그 사람 목소리로 소식이 자동 낭독된다
3. 큰 버튼으로 '좋아요' / '다시 듣기' 같은 최소 반응
4. 말로 답장: 마이크 버튼을 누르고 말하면 AI가 텍스트로 변환(STT)해 가족에게 전달된다

## 6. 왜 이렇게 설계했나 (핵심 의사결정)

### 6-1. 왜 서버가 두 개인가
생성 기능(아바타·보이스팩·낭독)은 전부 AI 모델이고, 그 모델은 **Python 생태계**다. Java에선 못 돌린다. 그래서:

- **familog-server (Java/Spring)** = 도메인·저장·상태관리를 맡는 **관리자**
- **familog-ai (Python/FastAPI)** = AI 모델을 돌려 결과물을 뱉는 **생성 공장**

둘은 HTTP로 대화한다. 이 분리는 억지가 아니라 "각자 잘하는 것을 맡긴다"는 실무적 설계다.

### 6-2. 왜 비동기로 처리하나
아바타·보이스팩·낭독 생성은 몇 초~몇십 초 걸린다. 이걸 동기로 처리하면 화면이 그동안 멈춘다. 시니어 서비스에서 화면 멈춤은 치명적이다.

그래서 **요청 즉시 "생성 중" 상태로 응답하고, 실제 생성은 백그라운드에서** 돌린다. 화면은 준비되면(READY) 표시한다. 이 덕분에 화면이 멈추지 않고, 생성이 실패해도 폴백으로 넘어갈 수 있다.

### 6-3. 비용 결정
- **보이스팩·낭독**: 오픈소스 모델(CosyVoice2)을 맥에서 직접 구동 → **무료.** "직접 서빙했다"는 기술적 어필도 된다.
- **아바타**: OpenAI 이미지 API 호출 → **유료(이미지당 소액).** 로컬 이미지 모델은 메모리 부담이 커서 외부로 뺐다.

## 7. 기대 효과

1. **디지털 포용 및 정서적 연결** — AI 음성 기반 소통으로 시니어의 디지털 접근성을 높이고, 가족 간 소통을 활성화하여 정서적 고립을 완화한다.
2. **AI 서비스 경쟁력 강화** — 에이닷(A.)의 음성 AI·AI 페르소나 기술 활용을 확대하여 AI 기반 커뮤니케이션 플랫폼 경쟁력을 강화한다.

## 8. 발표 어필 포인트

심사에서 강조할 인과 사슬:

> 시니어가 읽기 힘들다 → 들려주자 → 실제 가족 목소리가 필요 → 오픈소스 클로닝 모델(Python) → 그래서 추론 서버 분리 → 생성이 무거우니 비동기 → 그래서 상태관리로 로딩 없는 UX

즉 아키텍처(서버 분리·비동기)를 "기술 자랑"이 아니라 **"시니어 문제를 풀기 위한 필연적 선택"**으로 설명한다.

**데모 원칙**: 발표 중 실시간 생성은 지연·실패 리스크가 크니, 데모용 아바타·보이스팩·낭독은 미리 생성해 심어두고 시연한다. 자동 생성 로직은 살아있게 두되, 시연은 준비된 결과로.

**데모 시나리오 예시**
1. 손주: 사진 + "할머니 저 이번에 대회 나가요!" 업로드
2. 서버가 손주 보이스팩으로 낭독 생성
3. 시니어 화면: 손주 아바타 탭 → 손주 목소리로 자동 낭독
4. 임팩트 한 줄: "텍스트를 읽어주는 앱이 아니라, **가족의 얼굴과 목소리로 들려주는 앱**"

## 9. 개발 환경

- 백엔드: Java 17 / Spring Boot / MySQL (인텔리제이)
- AI 서버: Python / FastAPI (인텔리제이 + Python 플러그인 또는 파이참)
- 프론트: 별도 진행
- 개발 머신: MacBook M2, 16GB (TTS 로컬 구동, 아바타는 외부 API)
