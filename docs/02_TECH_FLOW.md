# Familog — 기술 플로우 (초안)

> 이 문서는 **구현 흐름과 순서** 중심이다. 구체 코드는 진행하며 바뀌므로,
> "무엇을 / 어떤 순서로 / 어떻게 흐르게" 만드는지를 잡는 용도.

---

## 1. 시스템 구성

```
[프론트] ──REST──▶ [familog-server :8080] ──HTTP──▶ [familog-ai :8000]
                   Java/Spring, MySQL            Python/FastAPI
                   도메인·상태·저장                AI 생성 전담
```

- **familog-server**: 멤버·게시물 관리, DB 저장, 상태관리, 프론트에 데이터 제공, AI 서버 호출.
- **familog-ai**: 생성 3종. `/avatar`(사진→캐릭터), `/voicepack`(목소리 등록), `/tts`(낭독).
- 둘 다 localhost. 배포 없이 로컬 데모.

**설계 3원칙**
1. 무거운 생성은 전부 **비동기 + 상태값**(PENDING→PROCESSING→READY→FAILED).
2. TTS/보이스팩 = 맥 로컬(무료), 아바타 = OpenAI(유료 소액).
3. 데모는 실시간 생성 대신 **미리 생성한 결과**로 시연.

---

## 2. 핵심 개념: 생성물 3종은 같은 패턴

아바타·보이스팩·낭독은 전부 **동일한 라이프사이클**을 탄다. 하나를 이해하면 셋 다 이해된다.

```
요청 도착
  → DB에 즉시 저장, 상태 = PENDING, 클라이언트엔 즉시 응답
  → 백그라운드에서 AI 서버 호출 (상태 = PROCESSING)
  → 성공: 결과 경로 저장 + 상태 = READY
     실패: 상태 = FAILED (폴백으로 전환)
  → 프론트는 상태를 폴링, READY면 표시/재생
```

| 생성물 | 트리거 | AI 엔드포인트 | 저장되는 것 | 상태 필드 |
|---|---|---|---|---|
| 아바타 PNG | 가입 시 | /avatar | 이미지 경로 | avatarStatus |
| 보이스팩 | 가입 시 | /voicepack | 보이스팩 식별자 | voiceStatus |
| 낭독 오디오 | 게시 시 | /tts | 오디오 경로 | ttsStatus |

**중요**: DB에는 결과물 자체가 아니라 **파일 경로/식별자만** 저장한다. 실제 파일(PNG·WAV)은 파일시스템에 두고, 화면은 그 경로로 불러온다.

## 3. 데이터 흐름 (엔드-투-엔드)

### 3-1. 가입 플로우
```
가족: 이름 + 사진 + 낭독음성 + 낭독대사 업로드
  → server: Member 저장 (avatarStatus=PENDING, voiceStatus=PENDING)
  → 즉시 memberId 응답 (프론트는 다음 화면으로)
  → [백그라운드] /avatar 호출  → 캐릭터 PNG 생성 → avatarUrl 저장, READY
  → [백그라운드] /voicepack 호출 → 보이스팩 저장 → voicepackId 저장, READY
```
아바타와 보이스팩은 **동시에 병렬로** 생성된다.

### 3-2. 게시 플로우
```
가족: 작성자 + 텍스트(+사진) 업로드
  → server: Post 저장 (ttsStatus=PENDING)
  → 즉시 postId 응답
  → [백그라운드] /tts 호출 (작성자의 voicepackId + 텍스트)
       → 그 사람 목소리로 낭독 오디오 생성 → ttsAudioUrl 저장, READY
```

### 3-3. 시니어 소비 플로우
```
홈: GET /members → 아바타(avatarUrl) 격자로 표시
게시물 탭: 그 글의 ttsAudioUrl 재생 (작성자 목소리)
  - ttsStatus=READY → 가족 목소리로 재생
  - PROCESSING → 플레이스홀더/표준 TTS로 먼저, 준비되면 교체
```

## 4. AI 서버 (familog-ai) 엔드포인트 명세

코드는 바뀌어도 **입출력 계약**은 유지한다. server는 이 계약만 알면 된다.

### /avatar
- **입력**: 사진 파일, member_id
- **처리**: OpenAI 이미지 API로 "3D memoji 스타일" 캐릭터 생성 (프롬프트 고정)
- **출력**: 저장된 PNG 경로
- 비고: 유료. 실제 붙일 때 OpenAI 이미지 API의 현재 엔드포인트/모델명 확인해 맞출 것.

### /voicepack
- **입력**: 낭독 음성 파일, 낭독 대사 텍스트, member_id
- **처리**: 참조 오디오 + 대사를 그대로 보이스팩으로 저장 (CosyVoice2는 참조 오디오 방식이라 임베딩 추출 불필요)
- **출력**: voicepack 식별자
- 비고: 가입 시 "지정 문장 낭독"이 곧 이 참조 오디오+대사가 된다.

### /tts
- **입력**: 텍스트, voicepack_id
- **처리**: 저장된 참조 오디오·대사로 CosyVoice2 zero-shot 낭독
- **출력**: 생성된 오디오 경로

> 세 엔드포인트의 인터페이스가 고정이면, 뒤에서 모델을 바꿔도(예: CosyVoice→OpenVoice, OpenAI→다른 이미지 API) server는 안 바뀐다.

## 5. 구현 순서 (플로우)

### 5-1. 병렬 원칙
세 파트(AI / 백엔드 / 프론트)를 동시에 굴린다. 특히 **백엔드는 AI 서버 완성을 기다리지 않는다.**
- 백엔드는 처음엔 **mock**으로 진행: AI 호출 자리에 "잠깐 뒤 READY + 더미 경로 반환" 스텁을 넣고 전체 흐름부터 완성.
- AI 서버가 준비되면 mock → 실제 호출로 교체.

### 5-2. AI 서버 순서
```
1. 서버 뼈대(/health) 떠서 응답 확인
2. /avatar (OpenAI) — 가볍고 즉시 테스트 가능 → 사진 1장으로 결과 확인
3. CosyVoice2 설치 + 낭독 "된다" 검증  ← 최우선 관문
4. /voicepack, /tts 엔드포인트화
```
**★ 3번(낭독 검증)이 프로젝트 전체의 첫 관문.** 팀 목소리 한 문장이 M2에서 낭독되는지 먼저 확인한다. 되면 TTS 확정, 안 되면(MPS 문제 등) CPU 폴백 또는 대체 모델 판단을 **즉시** 내린다. 오후까지 끌지 않는다.

### 5-3. 백엔드 순서
```
1. MySQL 실행 + DB 생성, Spring 부팅(8080) + DB 연결 확인
2. 엔티티(Member, Post) + 상태 enum + Repository
3. 비동기 활성화
4. AI 호출 클라이언트 (처음엔 mock)
5. 생성 서비스 (비동기 + 상태 전이)
6. 컨트롤러 (가입/게시/조회/폴링)
```

### 5-4. 통합
```
1. 두 서버 localhost 실연결
2. 가입 요청 → 멤버 조회 시 avatar가 READY로 바뀌는지
3. 게시 요청 → 게시물 폴링 시 tts가 READY + 오디오 재생되는지
4. 데모용 결과(아바타·보이스팩·낭독) 미리 생성해 심어두기
```

## 6. 상태·폴백 설계 (UX 직결)

각 생성물이 준비 안 됐을 때 화면이 무엇을 보여줄지 미리 정한다.

| 상태 | 아바타 | 낭독 |
|---|---|---|
| PENDING/PROCESSING | 기본 플레이스홀더 캐릭터 | 표준 TTS로 먼저 읽어주기 |
| READY | 생성된 캐릭터 PNG | 가족 목소리로 재생 |
| FAILED | 기본 캐릭터 유지 | 표준 TTS 유지 |

핵심: **어떤 상태에서도 화면은 뭔가를 보여준다.** 로딩으로 멈추는 순간이 없어야 한다.

## 8. 결정 기록 (왜 이렇게 정했나)

진행 중 내린 기술 선택과 그 이유. 발표에서 "왜 이 기술을 골랐나" 대응용.

### 8-1. 아바타는 "진짜 3D 모델"이 아니라 "3D풍 PNG"
- 회전 가능한 3D 메시(GLB) 아니고, **입체감 있게 렌더된 평면 이미지(PNG)**로 간다.
- 이유: 메인화면은 캐릭터가 격자로 뜨는 구조라 돌려볼 필요가 없다. 시니어가 캐릭터를 조작할 일이 없으므로 "3D처럼 보이는 이미지"면 충분하다.
- 진짜 3D 생성은 무겁고 하루엔 벅차며 시니어 서비스엔 과하다.

### 8-2. 아바타 스타일: Memoji/AR이모지 느낌 (정적)
- 레퍼런스: 애플 Memoji / 삼성 AR Emoji / 온젠가 같은 귀여운 캐릭터 룩.
- **실시간 표정 트래킹은 하지 않는다.** 온젠가는 카메라로 얼굴을 실시간 추적하지만, 우리 시니어는 "듣는" 사람이라 자기 얼굴을 트래킹할 일이 없다. 정적 캐릭터로 충분.
- 삼성/애플 공식 엔진은 외부에서 못 쓴다(폐쇄). 그래서 "그 느낌을 흉내내는 AI 변환"으로 간다.

### 8-3. 아바타 생성 = OpenAI 이미지 API
- 무료 후보(LightX/aiphotocraft 카툰 API, 오픈소스 이미지 모델 로컬)도 검토했으나:
    - 무료 카툰 API는 품질 편차가 커서 시니어 메인화면에 부적합.
    - 오픈소스 이미지 모델 로컬 구동은 M2 16GB 메모리 부담(TTS와 동시 로드 어려움).
- 그래서 품질·편의를 위해 **OpenAI 이미지 API(유료 소액)**로 결정. 프롬프트 고정으로 스타일 일관성 확보.
- 데모 분량이면 비용 미미. 이 부분만 무료 원칙에서 예외.

### 8-4. 보이스팩/낭독 = CosyVoice2 (폴백 OpenVoice)
- 유료 후보(ElevenLabs)는 무료 티어에서 보이스 클로닝이 막혀 있고(최소 유료 $5), "직접 서빙" 어필이 약해 제외.
- 무료 오픈소스 zero-shot 후보(CosyVoice2, OpenVoice, VoxCPM 등) 중:
    - **CosyVoice2**: 한국어 포함 다국어 화자 유사도 좋음, Apache-2.0, 참조 오디오 방식이라 구현 단순 → **1순위**.
    - **OpenVoice V2**: 세팅 가볍고 자료 많음 → 폴백.
- "가족의 실제 목소리로 들려준다"가 핵심 가치라 화자 유사도를 우선.

### 8-5. 무료 오픈소스 = "API 호출"이 아니라 "직접 서빙"
- CosyVoice2는 남의 API를 호출하는 게 아니라, **모델 가중치를 맥에 받아 우리 서버에서 직접 추론**한다.
- 그래서 "엘랩 API 갖다 썼다"가 아니라 "오픈소스 모델을 우리 인프라에 올려 보이스팩 파이프라인을 직접 구축했다"고 말할 수 있다 → 기술 어필 포인트.

### 8-6. 개발 환경 / IDE
- familog-server(Java) = 인텔리제이, familog-ai(Python) = 인텔리제이 Python 플러그인 또는 파이참.
- 웹스톰은 JS/TS 전용이라 Python(FastAPI) 개발 불가 → 프론트 몫.
- DB는 H2 대신 **MySQL** 사용(실 DB, 어필). 배포는 안 하고 로컬 데모.

### 8-7. 음성 메시지는 원본 유지, 재합성(STT→TTS) 안 함
- 녹음 메시지는 **원본 오디오 그대로** 전달하고, STT는 자막(텍스트) 생성용으로만 쓴다.
- 이유: ① 진짜 목소리(억양·감정)가 서비스 핵심 가치 — 합성이 원본을 못 이긴다. ② 이중 변환은 지연·실패 지점 2배. ③ STT 오인식이 음성으로 전파되는 위험(틀린 말을 그 사람 목소리로 하게 됨).
- 노이즈가 문제되면 재합성이 아니라 **노이즈 제거 후처리**(가벼운 필터)를 ai 서버에 한 단계 추가하는 것으로 대응. 해커톤 범위에선 생략.
- TTS(보이스팩)는 키보드로 입력한 텍스트 메시지에만 적용.

### 8-8. 현재 진행 상황 (세팅)
- familog-ai: 폴더+venv 생성, fastapi/uvicorn/python-multipart/openai 설치 완료.
- 다음: main.py 뼈대 → /avatar → CosyVoice2 낭독 검증 순.

### 8-9. CosyVoice2 M2 구동 검증 완료 (★관문 통과) — 2026-07-16
- **결과: 성공.** CosyVoice2-0.5B zero-shot 한국어 낭독이 M2 16GB에서 정상 생성됨 (Whisper 역전사로 문장 100% 일치 확인).
- **CPU로 동작** — CosyVoice 코드가 cuda/cpu만 분기(MPS 미지원). 성능: 모델 로드 ~2분(시작 시 1회), 생성 RTF 약 23 (5.7초 오디오에 130초). 비동기+미리 생성 전략(§1, §7)으로 충분히 커버.
- venv를 **Python 3.11**로 재구성 (기존 3.14는 torch 2.3.1 미지원). 설치 순서 함정(openai-whisper 빌드 시 torch 필요, setuptools<81)은 familog-ai/requirements.txt 주석에 기록.
- 모델 다운로드는 ModelScope가 ~1MB/s로 너무 느려 **HuggingFace 미러(hf_transfer)** 사용 (~4.5GB, 수 분 내 완료).
- `TTS_ENGINE=mock`(macOS say)으로 모델 없이 엔드포인트 계약만 테스트 가능 — server 팀은 mock으로 연동 개발 가능.

## 9. 이 문서에서 아직 안 정한 것 (진행하며 확정)

- 파일(사진 원본·오디오) 서빙 방식 (로컬 정적 리소스 vs 별도)
- 프론트 폴링 주기 / 실시간성 필요 여부
- 아바타 자동생성을 데모 파이프라인에 실제로 넣을지, 미리 생성으로 갈지
- 표준 TTS 폴백에 쓸 엔진
- 가족 그룹/초대 코드 등 커뮤니티 부가 기능 범위
